# BigQuery Table Guide
Last updated: 2026-06-16
Dataset: `angular-axle-492812-q4.qoyod_marketing`

Live table count verified by INFORMATION_SCHEMA query: **15 objects** (12 BASE TABLEs + 3 VIEWs)

---

## Full inventory

| Table | Type | What it contains | Why it exists separately | Can it merge? |
|---|---|---|---|---|
| `ads_daily` | BASE TABLE | One row per ad × day × channel. Spend, impressions, clicks, ad_id, ad_name, adset_id, adset_name, campaign_id, campaign_name, status, channel. Written by all paid-channel collectors. | Finest-grain spend table. The ad level is the atomic unit for pause/scale decisions — you cannot compute "this specific ad has spent $70 with 0 conversions" from campaign-level data. | Cannot merge into `campaigns_daily` because campaigns_daily is campaign-grain (one row per campaign per day) — collapsing ad rows into it would lose the ad_id/ad_name dimensions needed for creative-level analysis. Cannot merge into `wide_ads` because `wide_ads` is rebuilt FROM `ads_daily` (it is the source, not a duplicate). |
| `agent_activity_log` | BASE TABLE | One row per agent action: role, status (started/completed/failed), action_type, detail_json, timestamp. Written by `logs/activity_logger.py`. | Operational telemetry. Required for the Nexa Agent Activity Hex dashboard and for debugging scheduler failures post-hoc. | Cannot merge into any other table — it is an append-only log with a different schema (no date/channel/spend). Merging into `asana_task_status` would conflate agent telemetry with external task state. |
| `asana_task_status` | BASE TABLE | One row per Asana task GID: gid, name, status, assignee, due_date, created_at, last_synced. Written by `collectors/asana_sync.py`. | Makes Asana task state queryable from BQ so the agent can check "is this recommendation task still open?" without an API call in the middle of analysis. | Cannot merge into `agent_activity_log` — Asana tasks are external objects with lifecycle; the log is internal telemetry. Different schemas, different update cadence (Asana syncs every 6h; the log is append-only real-time). |
| `campaigns_daily` | BASE TABLE | One row per campaign × day × channel. Spend, impressions, clicks, campaign_id, campaign_name, channel, objective, status. Written by all 6 paid-channel collectors (Google, Meta, Snap, TikTok, LinkedIn, Microsoft). | Campaign-level spend is authoritative — PMax and awareness campaigns have no ad-level breakdown (PMax manages creative automatically; awareness buys are impression-optimised with no individual ad rows). Losing campaigns_daily means losing spend data for entire campaign types. | Cannot merge into `wide_ads` because wide_ads is ad-grain — a PMax campaign has no matching ad_id rows in ads_daily, so it would disappear from the JOIN. campaigns_daily must stay separate to capture all campaign spend including non-ad-grain types. |
| `connector_health_log` | BASE TABLE | One row per connector health check: connector name, check_type (token_valid, data_fresh, row_count, etc.), status (ok/warn/error), detail, timestamp. Written by `analysers/connector_tracker.py`. | Health monitoring. Lets the agent detect "LinkedIn token has been expired for 3 days" without querying the live API. Also powers the Databox connector-health monitors. | Cannot merge into `agent_activity_log` — different schema and different purpose. Health checks are structured pass/fail per connector; activity log is unstructured action telemetry. Merging would make both harder to query. |
| `ga4_sessions_daily` | BASE TABLE | GA4 web sessions by date, page_path, source/medium, device. Written by a GA4 collector. | Landing page performance requires session data independent of ad spend. GA4 sessions cover organic, direct, and email traffic that has no row in any paid-channel table — it would be lost if merged elsewhere. | Cannot merge into `organic_page_daily` — ga4_sessions_daily is at page_path grain (URL-level); organic_page_daily is at channel grain (FB/IG/YouTube post level). Different sources, different grains, different purposes. |
| `hubspot_deals_individual` | BASE TABLE | One row per HubSpot deal. deal_qoyod_source (channel), deal_utm_campaign, deal_utm_content, deal_utm_audience, deal_campaign_id_sync, stage, pipeline, amount_total (USD), amount_native (SAR), createdate, closedate, hs_object_id. Full-window mirror from 2025-01-01. | Record-level deal data is required for stable-ID attribution (deal_campaign_id_sync → campaigns_daily.campaign_id). Pre-aggregated tables like `hubspot_deals_daily` hide the record-level UTM fields needed to resolve campaign-level attribution. Also: deals span multiple stages and pipelines that require per-record tracking. | Cannot merge into `hubspot_leads_individual` — deals and leads are different HubSpot objects with different lifecycles. A deal starts after a lead qualifies. Merging would make it impossible to compute CPL (lead-based) separately from ROAS (deal-based). Column schemas are almost entirely different. |
| `hubspot_leads_individual` | BASE TABLE | One row per HubSpot lead. lead_utm_campaign, lead_utm_content, lead_utm_audience, qoyod_source, lead_status (New/Qualified/Disqualified/etc.), hs_object_id, createdate. Full-window mirror from 2025-01-01. | Source of truth for all leads including organic/direct/email. `wide_ads` only covers paid channels — joining leads into it would drop the ~30% of leads that arrive via organic/direct. The individual table is also needed for disqualification-rate calculations (requires knowing both leads_total and leads_disqualified per ad/adset) without pre-aggregating. | Cannot merge into `wide_ads` — wide_ads is paid-channel only; hubspot_leads_individual contains all leads. Merging would either drop non-paid leads or bloat wide_ads with organic rows that have no spend. Either outcome corrupts attribution. |
| `keywords_daily` | BASE TABLE | One row per keyword × day × channel (Google + Microsoft only). keyword_text, match_type, adgroup_id, adgroup_name, campaign_id, quality_score, impression_share, search_impression_share, lost_is_rank, lost_is_budget, clicks, impressions, spend, conversions. | Keyword-level data exists only for search channels (Google, Microsoft). It cannot be stored in `ads_daily` — ads_daily is ad-grain and a keyword is a different concept (a bid target, not a creative). Keyword policy enforcement (QS, IS, language-match, 30-cap rule) requires per-keyword rows. | Cannot merge into `ads_daily` — a keyword is a bid target; an ad is a creative unit. One adgroup can have 30 keywords but only 3 ads. Merging would either create a cartesian product (30×3 rows per day per adgroup) or lose one dimension entirely. `wide_keywords` is built FROM keywords_daily the same way wide_ads is built from ads_daily. |
| `organic_page_daily` | BASE TABLE | One row per day × channel × page/post for organic social (Facebook/Instagram pages, YouTube channel, LinkedIn organic). Impressions, reach, engagements, video_views. | Organic performance is tracked separately from paid because organic has no spend column and no lead attribution. It feeds the organic section of the Hex dashboard. | Cannot merge into `campaigns_daily` — campaigns_daily is paid-channel spend only; organic posts have no campaign_id. Merging would create NULL spend rows for organic posts that would corrupt CPL/CPQL calculations in any aggregation over the table. |
| `wide_ads` | BASE TABLE | Ad-grain denormalized table: one row per ad × day × channel with spend (from ads_daily) + leads/deals attributed to that ad (from hubspot_leads_individual + hubspot_deals_individual). Rebuilt every 6h by `materialize_heavy_views()`. | The single join surface for all paid-media performance analysis. Building the join once (rather than repeating a 3-way join in every Hex cell or Python query) is cheaper and faster. New queries should target wide_ads directly with GROUP BY rather than the reporting views. | Cannot merge into `campaigns_daily` — wide_ads is ad-grain; campaigns_daily is campaign-grain. PMax campaigns have no ad_id rows in wide_ads but DO appear in campaigns_daily. Replacing campaigns_daily with a roll-up from wide_ads would silently drop PMax spend. |
| `wide_keywords` | BASE TABLE | Keyword-grain denormalized table: one row per keyword × day × channel with spend + leads/deals attributed to that keyword. Rebuilt every 6h alongside wide_ads. | Same rationale as wide_ads but at keyword grain. Keyword-level CPQL requires keyword spend + attributed leads in one row. Built once, queried many times. | Cannot merge into `wide_ads` — keywords and ads are different grains and different dimensions. A keyword row has keyword_text, match_type, QS, IS. An ad row has ad_id, ad_name, creative details. Merging them would produce a misshapen table where half the columns are NULL for every row. |
| `hubspot_deals_daily` | VIEW | Day × pipeline × channel aggregation over `hubspot_deals_individual`. Columns: date, pipeline, qoyod_source, deals_created, deals_won, amount_total, amount_won. | Backward-compat shim — 40+ Python files and Hex cells reference `hubspot_deals_daily` by name. The underlying physical table was dropped 2026-06-15 and replaced by this VIEW over the individual table. Preserving the name means zero consumer changes. | Cannot be dropped until all consumers are migrated to query `hubspot_deals_individual` directly with GROUP BY. As of 2026-06-16, the VIEW has at least 40 consumers. Dropping it would break the reporting scheduler, the Flask health endpoint, and multiple Hex cells simultaneously. |
| `hubspot_leads_module_daily` | VIEW | Day × utm_campaign aggregation over `hubspot_leads_individual`. Columns: date, lead_utm_campaign, qoyod_source, leads_total, leads_qualified, leads_disqualified, leads_open. | Same backward-compat rationale as hubspot_deals_daily. The physical table was dropped 2026-06-15. This VIEW is the join target in the pre-aggregation CTE pattern enforced across all Python files (CRITICAL_KPI_RULES.md): `WITH hs AS (SELECT date, lead_utm_campaign, SUM(leads_total) FROM hubspot_leads_module_daily GROUP BY ...)`. | Cannot be dropped — it is referenced by name in 40+ Python files, the CRITICAL_KPI_RULES guard, and every Hex campaign cell. Dropping it before migrating all consumers would silently break CPL/CPQL calculations across the entire stack. |
| `v_keyword_performance` | VIEW | Keyword-grain performance view with QS, impression share, leads, and CPQL. Source: `wide_keywords` + `hubspot_deals_individual`. | Provides a single pre-joined query surface for the keyword audit (`scripts/audit.py keywords`) and the Google Ads Hex cells. Avoids repeating the wide_keywords + deals join in every call site. | Cannot be dropped while `analysers/google_ads_audit.py` and the keyword Hex cells reference it. It will be deprecated once those consumers are migrated to query `wide_keywords` directly — no timeline set as of 2026-06-16. |

## Layer summary (how the tables relate)

```
Layer 0 — Raw store (written by collectors every 6h)
  campaigns_daily, ads_daily, keywords_daily
  ga4_sessions_daily, organic_page_daily
  agent_activity_log, connector_health_log, asana_task_status

Layer 1 — Individual mirrors (full-window re-pull from 2025-01-01)
  hubspot_leads_individual, hubspot_deals_individual

Layer 2 — Wide joined tables (rebuilt every 6h from L0 + L1)
  wide_ads         ← ads_daily JOIN hubspot_{leads,deals}_individual
  wide_keywords    ← keywords_daily JOIN hubspot_{leads,deals}_individual

Layer 3 — Reporting views (GROUP BY rollups over L2, recreated every 6h)
  [not listed in INFORMATION_SCHEMA at time of query — recreated dynamically]
  paid_channel_daily, paid_channel_campaign_daily
  v_adset_performance, v_ad_performance

Layer 3b — Compat views (same name as dropped physical tables, source = L1)
  hubspot_leads_module_daily  ← hubspot_leads_individual
  hubspot_deals_daily         ← hubspot_deals_individual
  v_keyword_performance       ← wide_keywords
```

## Query guidance for new work

- **For new cells or scripts**: query `wide_ads` or `wide_keywords` directly with an explicit `GROUP BY`. Do not add new consumers to the reporting views — they exist for backward compat only.
- **For campaign-level spend including PMax**: query `campaigns_daily` directly. Do not try to roll up from `wide_ads` — PMax rows are absent at ad grain.
- **For leads/CPQL**: always pre-aggregate `hubspot_leads_module_daily` in a CTE before joining to spend. Direct JOIN without a CTE causes spend fan-out (see CRITICAL_KPI_RULES.md).
- **For deal revenue/ROAS**: use `hubspot_deals_individual` with `deal_qoyod_source` for channel attribution. UTM campaign attribution is ~66% coverage — always include an `(unknown)` bucket.
